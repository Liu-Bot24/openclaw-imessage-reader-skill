## 0. 这个技能做什么

让用户通过任何已接入的聊天渠道（飞书、微信等）用自然语言查询本机 iMessage / SMS / RCS 消息。

**典型用法：**
- "收短信" → 返回最近 30 分钟所有短信
- "ChatGPT 验证码是多少" → 返回含 ChatGPT/OpenAI 关键词的短信
- "95588 发来的消息" → 返回指定发送方的消息
- "最近 1 小时的 iMessage" → 返回指定时间范围和类型的消息

**返回字段：** 时间、消息类型（SMS/iMessage/RCS）、发送方号码、接收方号码、消息内容。

## 1. 安全架构

**核心原则：权限最小化。**

```
用户发消息 → OpenClaw gateway (node, 无 FDA)
           → imessage_reader.py (python, 无 FDA, 不接触 chat.db)
           → imessage-db-reader (compiled Swift binary, 有 FDA, 只读 chat.db)
           → JSON 输出 → Python 格式化 → 返回给用户
```

只有 `imessage-db-reader` 这一个二进制拥有完全磁盘访问权限。它无依赖、不联网、不写磁盘，源码可审计。node / python / Terminal 均无需 FDA。

## 4. 编译 Swift 二进制


## 7. 验证

> **重要：** 由于 macOS TCC 的 "responsible process" 机制，从终端直接运行二进制进行数据库查询会被拦截。不触库的操作（如 `--help`）不受影响。验证时请用 `launchctl submit` 模拟 LaunchAgent 环境。

### 7.1 验证 Swift 二进制

```bash
rm -f /tmp/imessage-test-out.txt /tmp/imessage-test-err.txt

launchctl submit -l ai.openclaw.imessage-test \
  -o /tmp/imessage-test-out.txt \
  -e /tmp/imessage-test-err.txt \
  -- ~/.openclaw/workspace/scripts/imessage-db-reader --minutes 1440 --limit 3

sleep 5

echo "=== stdout ==="
cat /tmp/imessage-test-out.txt
echo "=== stderr ==="
cat /tmp/imessage-test-err.txt

launchctl remove ai.openclaw.imessage-test
```

**预期结果：** stdout 应输出 JSON 数组（含消息或空数组 `[]`）。stderr 应为空。如果 stderr 包含 `Full Disk Access required`，说明第 6 步没做好，让用户检查。

### 7.2 验证完整调用链（Python → Swift）

```bash
rm -f /tmp/imessage-test2-out.txt /tmp/imessage-test2-err.txt

launchctl submit -l ai.openclaw.imessage-test2 \
  -o /tmp/imessage-test2-out.txt \
  -e /tmp/imessage-test2-err.txt \
  -- /usr/bin/python3 ~/.openclaw/workspace/scripts/imessage_reader.py --minutes 1440 --limit 3

sleep 8
echo "=== stdout ==="
cat /tmp/imessage-test2-out.txt
echo "=== stderr ==="
cat /tmp/imessage-test2-err.txt
launchctl remove ai.openclaw.imessage-test2
```

**预期结果：** 应输出格式化的消息列表，类似：
```
共 N 条消息：

[1] 2026-03-24 23:35:29  SMS
    发送方: 10690618999942244895
    接收方: +8613800001234
    内容: 【稀宇科技】您好，您登陆的验证码：709101...
```

### 7.4 验证 OpenClaw 实际调用

当 7.1 ~ 7.3 都通过后，通过任意已接入渠道发送：

- "收短信"
- "看看最近的验证码"
- "最近 1 小时有什么消息"

**如果 OpenClaw 回复"收到"但不执行脚本**，逐层排查：

1. **技能未加载** — `openclaw skills info imessage-reader` 确认，必要时 `openclaw gateway restart`
2. **exec 审批拦截** — `openclaw approvals get --json` 检查，首次执行弹审批时手动放行
3. **渠道级工具策略禁止 exec** — `openclaw doctor` 全局检查，或对比已知正常渠道的配置
4. **大模型没匹配意图** — 试更明确的表述如"用 imessage-reader 技能查最近的短信"，或开新对话

## 8. 故障排查

| 问题 | 原因 | 解决 |
|------|------|------|
| `Full Disk Access required` | 二进制没有 FDA | 重新执行第 6 步 |
| 终端直接运行报权限错，launchctl 正常 | macOS TCC responsible process 机制，正常现象 | 不影响使用 |
| Python 报 `imessage-db-reader not found` | 二进制没复制到目标路径 | 重新执行第 5 步 |
| OpenClaw 说"收到"但不执行 | 四层原因逐层查 | 见 7.4 |
| 输出为空 `[]` | 时间范围内无消息 | 加大 `--minutes`，如 1440 |
| `database is locked` | Messages.app 持锁，暂时性 | 重试即可，二进制已内置 3 秒等待 |
| `failed to parse reader output` | 临时文件冲突，暂时性 | 重试即可 |
| `launchctl submit failed` | launchctl 提交任务失败 | 检查是否有同名 label 残留：`launchctl remove <label>` |
